# Samato Microservice Bible 🍔

A complete reference microservice system designed to be your **interview preparation bible** for backend / platform engineering roles.

> **Project:** Samato
> **Domain:** Food delivery (like Uber Eats / DoorDash)
> **Why this domain:** Exercises nearly every microservice concern in one realistic flow — gateways, sagas, async events, distributed transactions, caching, resilience, observability, security, and more.

## Quick start

For the long version (prerequisites, troubleshooting, what to do when things break), see [`RUN-THE-BIBLE.md`](RUN-THE-BIBLE.md). The minimum:

```bash
# 1. Prerequisites: Java 21, Maven 3.9+, Docker Desktop
java -version && mvn -version && docker --version

# 2. Build the jars (Maven 3.9+ required for Spring Boot 3.3.4)
mvn -B -DskipTests package

# 3. Build the docker images from the local jars
cd infra && docker compose build

# 4. Bring up the full stack
docker compose up -d
# wait 60-90s for Spring contexts to start
```

Verify:
- Eureka dashboard: <http://localhost:8761> (7 services registered)
- API Gateway:    <http://localhost:8080/actuator/health>
- Auth (JWKS):    <http://localhost:9000/.well-known/jwks.json>
- Schema Registry: <http://localhost:8085/subjects>
- Kafka UI:       <http://localhost:8091>  (port 8091 — 8081 is user-service)
- Prometheus:     <http://localhost:9090>
- Grafana:        <http://localhost:3000> (admin / admin)
- Zipkin:         <http://localhost:9411>

The end-to-end happy path (register → login → place order → saga runs → payment attempts) is in [`RUN-THE-BIBLE.md` §4](RUN-THE-BIBLE.md).

## What's in this repo

### Services (9 total, all on Java 21 + Spring Boot 3.3.4)

| Service | Port | Pattern demonstrated |
|---|---:|---|
| **api-gateway** | 8080 | Spring Cloud Gateway, JWT validation, rate limit, route config |
| **config-service** | 8888 | Spring Cloud Config Server, git-backed config repo |
| **discovery-service** | 8761 | Netflix Eureka registry |
| **auth-service** | 9000 | Spring Authorization Server, JWT issuance, JWKS endpoint |
| **user-service** | 8081 | Profile management, role-based access |
| **restaurant-service** | 8082 | Restaurant + menu catalog, OpenSearch index |
| **search-service** | 8087 | CQRS read model, OpenSearch + Redis cache |
| **order-service** | 8083 | Saga orchestrator, outbox, idempotency, Feign clients |
| **payment-service** | 8084 | **Event-sourced** ledger, Razorpay integration, HMAC webhook |

> Note: AUTH port. `auth-service` runs on **9000** (Spring Authorization Server's default). The gateway validates JWTs by hitting `auth-service:9000/.well-known/jwks.json` — there is no Keycloak in this build.

### Infrastructure

| Component | Version | Purpose |
|---|---|---|
| PostgreSQL | 16 | One database per service (`payment_service`, `order_service`, …) |
| Apache Kafka | 3.7 | Async messaging, outbox fanout, schema-registry companion |
| Redis | 7 | Cache-aside, rate-limit tokens, distributed locks |
| OpenSearch | 2.x | Search index for restaurant catalog |
| Zipkin | latest | Distributed tracing UI |
| Prometheus | latest | Metrics scrape |
| Grafana | latest | Dashboards (admin/admin, datasource pre-provisioned) |

### Shared modules

- **`shared/`** — cross-cutting code: `DomainException`, `GlobalExceptionHandler`, `CorrelationIdFilter`, `MdcKeys`. Every service depends on this.
- **`shared-kafka/`** — Avro schemas for cross-service events. The Java classes are generated by the Avro Maven plugin. Three schemas today: `PaymentChargedEvent`, `PaymentRefundedEvent`, `PaymentFailedEvent`.

## What this project is (and isn't)

**It is:** a structural reference that is **verified up and running end-to-end** on this machine. `mvn package` builds all 12 modules. `docker compose up -d` brings up 18 containers; every service endpoint returns HTTP 200; 7 services are registered in Eureka. The first commit that includes the bring-up is on `main`.

Every service follows Spring Boot conventions, has a designer-notes doc, and exercises a real microservice pattern. Good for interviews, good for onboarding, good as a starting point.

**It isn't (yet):** a CI-tested system. There are no integration tests, and the live end-to-end saga (a real `POST /api/orders` through the gateway that triggers the saga and a Razorpay call) has not been executed — only the standalone code-path smoke test. The Razorpay SDK calls will use placeholder keys unless you provide real test keys via env. Phases 6+ (delivery, notification, analytics, hardening) are still pending.

**Honest plan to get to a CI-tested system:** the next step is `RUN-THE-BIBLE.md` §10 (Testcontainers) — add an integration-test module that brings up the same stack in CI and exercises the saga. That's the path to "the build is green, the saga ran, the money moved."
